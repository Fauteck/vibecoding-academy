Wann brauche ich diese Seite?

Diese Seite ist für dich relevant, wenn:

  • dein Prototyp bereits sichtbar funktioniert
  • du ihn lokal sauber starten willst
  • du ihn anderen zeigen möchtest
  • du ihn öffentlich oder halböffentlich bereitstellen willst

Überblick

Für erste Vibecoding-Projekte sind meist drei Wege relevant:

  1. Lokal ausführen – Gut zum Testen auf dem eigenen Rechner
  2. GitHub Pages – Gut für einfache statische Projekte und schnelle öffentliche Demos
  3. Docker – Gut, wenn du reproduzierbare Umgebungen oder flexibleres Deployment brauchst

1. Lokal ausführen

Ziel

Du willst dein Projekt erst einmal zuverlässig auf deinem eigenen Rechner starten und testen.

Typischer Ablauf

  • Projektordner öffnen
  • Abhängigkeiten installieren
  • Lokalen Entwicklungsserver starten
  • Anwendung im Browser testen

Darauf achten

  • Gibt es eine Startanweisung?
  • Werden Umgebungsvariablen benötigt?
  • Sind Fehler im Terminal sichtbar?
  • Funktioniert die App auch nach einem Neustart?
Empfehlung

Nutze lokal zuerst einen einfachen, reproduzierbaren Ablauf. Dokumentiere kurz: wie das Projekt gestartet wird, welche Voraussetzungen nötig sind und welche Befehle wichtig sind.

2. GitHub Pages

Geeignet für

  • Einfache statische Seiten
  • Frontend-Prototypen
  • Klickbare Demos
  • Workshop-Ergebnisse ohne serverseitige Logik

Weniger geeignet für

  • Projekte mit Backend
  • Anwendungen mit Logins
  • Projekte mit sensiblen Daten
  • Komplexe serverseitige Verarbeitung

Grundidee

Du veröffentlichst ein statisches Build deines Projekts über ein GitHub-Repository.

Typischer Ablauf

  1. Projekt in ein GitHub-Repository legen
  2. Prüfen, ob das Projekt statisch deploybar ist
  3. Build erzeugen
  4. GitHub Pages aktivieren
  5. Veröffentlichte URL testen
Vor dem Veröffentlichen prüfen

Sind sensible Daten ausgeschlossen? Sind API-Keys entfernt? Stimmen Pfade und Assets? Funktioniert die Anwendung auch auf einer Unterseite bzw. GitHub-Pages-URL?

Praktischer Hinweis: GitHub Pages ist ideal, wenn du schnell eine öffentliche Demo brauchst und dein Projekt keine serverseitigen Funktionen benötigt.

3. Docker

Geeignet für

  • Reproduzierbare Ausführung
  • Standardisierte Entwicklungs- oder Testumgebungen
  • Spätere Übergabe an andere Systeme
  • Projekte, die nicht nur rein statisch sind

Grundidee

Docker verpackt deine Anwendung in eine definierte Umgebung, die auf anderen Systemen leichter reproduzierbar läuft.

Typischer Ablauf

  1. Dockerfile erstellen oder prüfen
  2. Container bauen
  3. Container lokal starten
  4. Ports und Umgebungsvariablen korrekt setzen
  5. Anwendung testen

Vor dem Einsatz prüfen

  • Welche Dateien müssen ins Image?
  • Welche Variablen dürfen nicht ins Image?
  • Welche Ports werden benötigt?
  • Was muss persistent gespeichert werden?
  • Ist Docker für dieses Projekt überhaupt nötig?
Empfehlung

Docker ist oft sinnvoll, wenn du mehr Kontrolle über die Laufzeitumgebung brauchst. Für sehr einfache Frontend-Demos ist GitHub Pages meist schneller.

Welcher Weg passt zu meinem Projekt?

Weg Nutze es, wenn du …
Lokal noch testest, schnell iterierst oder dein Projekt noch nicht teilen musst
GitHub Pages eine einfache öffentliche Demo willst, ein statisches Frontend hast oder ohne Backend arbeitest
Docker reproduzierbare Umgebungen brauchst, mehr technische Kontrolle willst oder dein Projekt flexibler betreiben möchtest

Vor jeder Veröffentlichung prüfen

Bevor du etwas veröffentlichst, gehe diese Fragen durch:

  • Funktioniert der Hauptanwendungsfall?
  • Gibt es offensichtliche Fehler?
  • Sind API-Keys und sensible Daten entfernt?
  • Sind verwendete Bilder, Icons und Texte rechtlich unkritisch?
  • Ist klar, dass es sich um einen Prototyp handelt?
  • Brauchst du Impressum oder Datenschutz-Hinweise?

Häufige Fehler beim Veröffentlichen

  • Versehentlich sensible Daten im Repository
  • Deployment eines Projekts, das nicht statisch geeignet ist
  • Falsche Pfade bei GitHub Pages
  • Nicht dokumentierte Startschritte
  • Ungeprüfte Umgebungsvariablen
  • Fehlender letzter Funktionstest

Empfehlung für Workshop-Projekte

Für die meisten Workshop-Ergebnisse gilt:

  • Zuerst lokal prüfen
  • Bei einfachen Frontends GitHub Pages bevorzugen
  • Docker nur nutzen, wenn es einen klaren Mehrwert gibt

Das Ziel ist nicht maximale Infrastruktur, sondern ein sauber demonstrierbares Ergebnis.
